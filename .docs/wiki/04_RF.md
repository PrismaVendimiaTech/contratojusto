# 04 - Requerimientos Funcionales: ContratoJusto

## Decision Priority
Velocidad de entrega > Demo funcional > Completitud

## Convenciones
- **RF-XX-YY**: RF del area XX, numero YY
- **Prioridad**: P0 = demo path, P1 = importante, P2 = nice to have
- **Trazabilidad**: cada RF referencia su FL de origen

## RF-01: Gestion de Contratos

### RF-01-01: Crear contrato laboral
- **FL**: FL-01
- **Prioridad**: P0
- **Descripcion**: El empleador puede crear un contrato laboral definiendo: address del trabajador, porcentaje de ahorro (savings_pct), porcentaje de indemnización (severance_pct), y address del token USDC.
- **Precondiciones**: Wallet conectada, employer tiene XLM para gas
- **Input**: employer: Address (implícito, se toma de la wallet conectada), worker: Address, token: Address, savings_pct: u32, severance_pct: u32
- **Output**: Contrato inicializado en Soroban, contract ID retornado
- **Reglas**: savings_pct + severance_pct == 100, ambos > 0
- **Errores**: E-01-01: Porcentajes no suman 100. E-01-02: Wallet no conectada. E-01-03: XLM insuficiente para gas.
- **Criterio de aceptacion**: Despues de crear, get_info() retorna los parametros correctos y balances en 0.

### RF-01-02: Consultar contrato
- **FL**: FL-03 (parcial)
- **Prioridad**: P0
- **Descripcion**: Cualquier usuario puede leer el estado completo del contrato (balances, parametros, estado).
- **Input**: Ninguno (read-only, contract ID en config)
- **Output**: ContractInfo con todos los campos
- **Criterio de aceptacion**: get_info() y get_balance() retornan datos actualizados despues de cada operacion.

### RF-01-03: Terminar contrato
- **FL**: FL-05
- **Prioridad**: P0
- **Descripcion**: El empleador puede terminar el contrato. Al terminar, la indemnización acumulada se transfiere automáticamente al trabajador.
- **Precondiciones**: Contrato activo (is_terminated == false), wallet conectada como employer
- **Input**: employer address (require_auth)
- **Output**: is_terminated = true, severance transferido a worker
- **Reglas**: Solo employer. Irreversible. Auto-libera severance.
- **Errores**: E-01-04: Ya terminado. E-01-05: No es employer.
- **Criterio de aceptacion**: Despues de terminar, is_terminated == true, severance_balance == 0, worker recibio USDC.

## RF-02: Depósitos

### RF-02-01: Depositar USDC
- **FL**: FL-02
- **Prioridad**: P0
- **Descripcion**: El empleador deposita USDC en el contrato. El contrato separa automáticamente: savings_pct% va a pool ahorro, severance_pct% va a pool indemnización.
- **Precondiciones**: Contrato activo, employer tiene USDC suficiente + trustline + XLM gas
- **Input**: from: Address (employer), amount: i128 (> 0)
- **Output**: savings_balance += amount * savings_pct / 100, severance_balance += remainder
- **Reglas**: Solo employer. amount > 0. Contrato no terminado. Rounding: severance = amount - savings (evita perdida por redondeo).
- **Errores**: E-02-01: Monto 0 o negativo. E-02-02: No es employer. E-02-03: Contrato terminado. E-02-04: USDC insuficiente. E-02-05: Trustline falta.
- **Criterio de aceptacion**: Despues de depositar 100 USDC con 70/30 split, savings_balance == 70, severance_balance == 30.

## RF-03: Reclamos

### RF-03-01: Reclamar ahorro
- **FL**: FL-04
- **Prioridad**: P0
- **Descripcion**: El trabajador puede reclamar todo el ahorro acumulado en cualquier momento.
- **Precondiciones**: savings_balance > 0, wallet conectada como worker
- **Input**: to: Address (worker, require_auth)
- **Output**: USDC transferido de contrato a worker, savings_balance = 0
- **Reglas**: Solo worker. Reclama TODO el savings, no parcial (MVP).
- **Errores**: E-03-01: No hay ahorro. E-03-02: No es worker.
- **Criterio de aceptacion**: Despues de reclamar, savings_balance == 0, worker tiene los USDC.

### RF-03-02: Indemnización automática al terminar
- **FL**: FL-05, FL-06
- **Prioridad**: P0
- **Descripcion**: Al terminar el contrato (RF-01-03), la indemnización se libera automáticamente al trabajador. No requiere acción separada del worker.
- **Input**: N/A (se ejecuta dentro de terminate())
- **Output**: severance_balance transferido a worker, severance_balance = 0
- **Criterio de aceptacion**: Despues de terminate(), worker tiene los USDC de indemnización.

## RF-04: AI Chat

### RF-04-01: Consultar balance via chat
- **FL**: FL-03
- **Prioridad**: P0
- **Descripcion**: El trabajador puede preguntar en castellano por su balance y el AI responde con datos reales del contrato on-chain.
- **Input**: Mensaje en texto libre ("cuanto tengo?", "mi patrón deposito?")
- **Output**: Respuesta en castellano con datos de get_balance(): ahorro, indemnización, total, cantidad de depósitos
- **Tool**: consultarBalance() -> llama Soroban get_balance() via Stellar SDK
- **Errores**: E-04-01: micro-proxy timeout -> "Disculpa, no puedo consultar en este momento"
- **Criterio de aceptacion**: Al preguntar "cuanto tengo?", AI responde con los montos correctos del contrato.

### RF-04-02: Consultar estado via chat
- **FL**: FL-03
- **Prioridad**: P1
- **Descripcion**: El trabajador puede preguntar por el estado general del contrato (activo/terminado, porcentajes, info del empleador).
- **Tool**: consultarEstado() -> llama Soroban get_info() via Stellar SDK
- **Criterio de aceptacion**: Al preguntar "como esta mi contrato?", AI describe el estado completo.

### RF-04-03: AI prepara reclamo
- **FL**: FL-07
- **Prioridad**: P0
- **Descripcion**: El trabajador puede pedir al AI que prepare una transacción de reclamo. El AI construye el XDR y lo presenta para firmar con la wallet conectada.
- **Input**: Mensaje tipo "quiero reclamar mi ahorro" o "saca mi plata"
- **Output**: AI responde con resumen + boton "Firmar transacción" que abre la wallet para confirmar
- **Tool**: prepararReclamo(tipo: 'ahorro') -> construye XDR de invoke claim_savings
- **Nota**: indemnización se libera automáticamente en terminate(), no requiere tool separada.
- **Reglas**: AI verifica balance antes de preparar tx. Si balance == 0, informa sin preparar tx.
- **Errores**: E-04-02: Balance 0 -> AI informa. E-04-03: XDR error -> AI sugiere usar botones manuales.
- **Criterio de aceptacion**: Al pedir "quiero mi ahorro", AI muestra monto + boton firmar. Al firmar, USDC llega al worker.

### RF-04-04: AI prepara depósito
- **FL**: FL-07
- **Prioridad**: P1
- **Descripcion**: El empleador puede pedir al AI que prepare un depósito.
- **Tool**: prepararDepósito(monto: number) -> construye XDR de invoke deposit()
- **Criterio de aceptacion**: Al pedir "deposita 100 USDC", AI muestra resumen + boton firmar.

## RF-05: Wallet

### RF-05-01: Conectar Wallet (multi-wallet)
- **FL**: Todos
- **Prioridad**: P0
- **Descripcion**: El usuario puede conectar su wallet desde la web app usando Stellar Wallets Kit modal que soporta multiples wallets compatibles. La conexion persiste durante la sesion.
- **Input**: Click "Conectar wallet"
- **Output**: Public key del usuario visible en la UI, ready para firmar txs
- **Errores**: E-05-01: Ninguna wallet compatible instalada -> mostrar instrucciones. E-05-02: Testnet no seleccionada -> mostrar instrucciones.
- **Criterio de aceptacion**: Al conectar, la UI muestra la public key truncada y permite firmar.

### RF-05-02: Firmar transacciónes
- **FL**: FL-01, FL-02, FL-04, FL-05, FL-07
- **Prioridad**: P0
- **Descripcion**: Toda operacion que modifique estado on-chain requiere firma via wallet conectada (Stellar Wallets Kit). El frontend construye la tx, la wallet la muestra, el usuario decide si firma.
- **Input**: XDR de la transacción
- **Output**: Tx firmada enviada a Stellar testnet
- **Criterio de aceptacion**: Al firmar, la tx se ejecuta exitosamente en testnet.
